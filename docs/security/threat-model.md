# Threat model

## Assets

| Asset | Sensitivity |
|---|---|
| Per-user Atlassian access tokens | Critical — full user-level Atlassian access until expiry. |
| Per-user Atlassian refresh tokens | Critical — long-lived user-level access (90d sliding). |
| Per-user Atlassian API tokens (side-channel) | Critical — same as above, generated by the user at id.atlassian.com. |
| `TOKEN_ENCRYPTION_KEY` | Critical — decrypts every above. |
| `GOJIRA_ORG_ADMIN_TOKEN` | **Catastrophic** — org-wide admin authority on `admin.atlassian.com`. |
| MCP access tokens | Limited — 1-hour TTL, revocable, opaque. |
| MCP refresh tokens | Limited — 30-day TTL, rotating, reuse-detected. |
| Operation journal | Sensitive — describes admin changes (good for audit; bad if leaked alongside the targets). |
| `ATLASSIAN_OAUTH_CLIENT_SECRET` | High — allows acting as the gojira-mcp OAuth client. |

## Adversaries considered

| Adversary | Threat |
|---|---|
| External attacker over the public network | Can hit `/health`, `/.well-known/*`, `/register`, `/authorize`, `/oauth/atlassian-callback`. Cannot reach `/mcp` without a valid bearer. |
| Compromised MCP client | Can replay its own bearer until it expires, can request new bearers via /authorize but only against scope they were granted. |
| Stolen MCP refresh token | Detected on second use via D1 reuse detection; whole family revoked. |
| Stolen Atlassian token (upstream) | Outside gojira's threat boundary — Atlassian's own revocation applies. |
| Stolen `TOKEN_ENCRYPTION_KEY` alone | Useless without Redis snapshot. |
| Stolen Redis snapshot alone | Encrypted blobs unreadable without the key. |
| Stolen both | Total user-credential disclosure. Recovery: rotate the key (forces re-auth across all users) + revoke all Atlassian tokens. |
| Compromised insider with the gojira-mcp host shell | Has access to env vars including the encryption key; can read Redis; effectively full compromise. Use host hardening + secret-manager pull-at-start to reduce exposure. |
| Malicious tool input (prompt injection) | The model could attempt to call destructive tools. Mitigations: D5 commit-positive consent (model must opt in to mutation); identity binding (caller fields can't be spoofed); D4 site pinning (can't hit a different tenant); D2 journal (every change is recorded). |

## Threat-model claims

### Identity binding cannot be spoofed by clients

The bearer's `accountId` extra-claim is *not* derived from any
client-supplied field. It came from `api.atlassian.com/me` called with
the user's own Atlassian access token during the OAuth callback. The
bearer is opaque (32-byte hex), stored in Redis, and validated by
`verifyAccessToken` on every `/mcp` call.

Tools never accept a caller/requester field from the client. Update
operations strip `id`, `key`, `creator`, `created`, `updated`, etc. via
`sanitizeIssueUpdate`.

### Site pinning blocks tenant confusion

`ATLASSIAN_PINNED_CLOUD_ID` is enforced at three points:

1. During callback: if the pinned id is not in the caller's
   accessible-resources, the OAuth flow fails with `invalid_grant`.
2. At call time: `resolveCloudId` returns the pinned id and refuses
   when the caller no longer has it accessible.
3. For API-token tools: the bound token's `cloud_id` must match the
   pinned value.

An attacker with a sandbox-grant cannot reach prod through the
prod-pinned instance, even if the model tries.

### Refresh-token reuse is detected

See [refresh-token-rotation.md](../architecture/refresh-token-rotation.md)
and [refresh-reuse.md](refresh-reuse.md). Replay of a rotated-away RT
triggers full-family revocation + a structured warn log + optional
webhook.

### admin_org caller-verification ratchets up

Even with `GOJIRA_ENABLE_ORG_ADMIN=true`, every `admin_org` tool call first
asserts the **calling user is themselves an org admin**. A non-admin
caller — even one whose bearer reaches the dispatch path — still cannot
dispatch `admin_org` tools because the runtime check fails.

### Operator allowlist wins

`GOJIRA_ENABLED_GROUPS` filters at registration time and at dispatch
time. Tools whose group isn't listed are never registered, and (if one
somehow leaked through) the dispatch wrapper rejects the call.

### Fail-closed identity, fail-open availability

| Layer | Stance |
|---|---|
| `requireBearerAuth` | Fail closed. No bearer → no `/mcp` access. |
| Tool dispatch | Fail closed. Bad scope → `INSUFFICIENT_PERMISSIONS`. |
| Site pinning | Fail closed. Cloud mismatch → refuse. |
| Org-admin verification | Fail closed. Non-admin → refuse. |
| Rate limiter | Fail open. Redis errors return `allowed=true` (with a warn log). Availability > correctness for this subsystem. |
| Audit sink | Fail open. Write failures log warn and continue. |

## Out of scope

- **Atlassian-side abuse.** A user with valid Atlassian admin rights can
  do destructive things via Atlassian's own UI; gojira-mcp doesn't try
  to be a policy engine on top of that.
- **Confidentiality of MCP wire traffic.** TLS is the operator's
  responsibility (Caddy overlay or native TLS); plain HTTP behind a
  proxy is acceptable in private networks but not exposed publicly.
- **Denial-of-service.** The per-user rate limiter slows abusive
  callers but the public-facing endpoints (`/health`, `/.well-known/*`,
  `/register`, `/authorize`) are not specifically protected. Front
  with a CDN or rate-limit-aware proxy if hostile traffic is expected.

## Residual risks

- **Per-user token theft from Redis.** With both the key and Redis a
  user's Atlassian tokens are recoverable. Mitigation: secret manager
  for the key + Redis access control + at-rest encryption of the host
  disk.
- **Model misbehaviour.** The model could spam destructive `commit:true`
  calls. The journal records every change for forensics; reverters
  cover the easy half; manual restoration is required for the rest.
- **Org-admin token compromise.** If the org-admin token leaks, an
  attacker has org-wide control until you revoke it. Mitigation:
  isolate `admin_org` to its own instance; minimize the number of humans
  with access to the env file; rotate regularly.
